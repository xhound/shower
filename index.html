Что такое UI-Kit
и зачем он нужен?

Решаем задачу

  1. Быстро собирать интерфейс, используя рабочие, протестированные, компоненты.
  2. Легко модифицировать внешний вид компонента.
  3. Легко расширять функциональность базовых UI-компонентов.

Решение!

UI-кит — библиотека компонентов, из которых мы быстро собираем, необходимые нам, элементы интерфейса, набивая их логикой взаимодействия с пользователем.

Бонус:

  1. UI-элементы выглядят так как надо на всех платформах.
  2. UX одинаков, и не требует длительного обучения для работы с приложением, при переходе с платформы на платформу.

CSS — боль на больших проектах

  1. Глобальное пространство имен
  2. Разрешение зависимостей
  3. Поиск «мертвого» кода
  4. Отсутствие констант
  5. Неоднозначный результат (каскад)

БЭМ

  • Блоки могут использоваться в нескольких местах сайта;
  • Элементы являются частью блока и не имеют функционального смысла вне блока;
  • Модификаторы представляют собой свойства блока или элемента, которые меняют его внешний вид или поведение.

БЭМ — не все так гладко

  1. Процедура выбора className очень кропотлива и отнимает достаточно большое количество времени.
  2. Длинные имена классов раздувают разметку.
  3. Необходимо явно расширять каждый компонент интерфейса при каждом повторном использовании.

БЭМ — не все так гладко

CSS-модули

  1. Динамическое создание имен классов для каждого локально заданного стиля.
  2. При помощи CSS-модулей можно автоматизировать БЭМ-нотацию, генерируя уникальные className.

Что делать?

Решение!

Создаем компонент для которого в последствии через пропсы будем задавать тему:

Button.js

				@themr('MyThemedButton')
				class Button extends Component {
				static propTypes = {
				...
				     theme: {
				        button: React.PropTypes.string,
				        icon: React.PropTypes.string,
				        content: React.PropTypes.string
				     }
				}
				render() {
				     const { theme, icon, children } = this.props;
				     return (
				         <button className={theme.button}>
				              { icon && <i className={theme.icon}>{icon}</i> }
				              <span className={theme.content}>{children}</span>
				         </button>
				      )
				}

SuccessButton.js

				import Button from './Button';
				import successTheme from './SuccessButton.css';
				export const SuccessButton = (props) => (
				     <Button {...props} theme={successTheme}>Yeah!</Button>
				);

Компонент Button один и тот-же! Мы просто меняем тему для него...

react-css-themr

Библиотека react-css-themr облегчает работу с темизируемыми компонентами.Можно определить тему не только для компонента, но и для всего приложения передавая компонентам, при помощи провайдера тем, соответствующие им темы через контекст.

Давайте сделаем компонент Stepper:

Его код выглядит примерно так:

		render() {

Переиспользуемые компоненты

Один компонент — множество способов использования. Круто!

HOC. Зачем?

Используя этот паттерн мы можем создать фабрику компонентов, гененрируя новые компоненты с расширенной функциональностью. А также можем отделить логику компонента от способов ее реализации.

Что дальше?

У нас есть библиотека гибких переиспользуемых, темизируемых компонентов! Нужен дефолтный набор компонентов попадающих в приложение.

Хочется:

Определить глобальный объект дефолтных компонентов 

	{
	      BUTTON: defaultButton,
	      INPUT: defaultInput,
	           ...
	      CHECKBOX: defaultCheckbox
	}

...и использовать его в приложении
